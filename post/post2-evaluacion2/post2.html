Entrega II: Modelización de Procesos y Datos para CryptoPulse

Publicado por Isaac Díaz Mora | Septiembre 2025 | Asignatura: Sistemas de Información I

Esta segunda entrega documenta las decisiones clave tomadas para la modelización del sistema CryptoPulse, cubriendo la metodología de desarrollo elegida, los diagramas de flujo de datos (DFD) y el diseño de la arquitectura de datos (Diagrama Entidad-Relación).

1. Metodología de Desarrollo: RAD con Prototipos

Debido a la naturaleza cambiante del mercado cripto y la necesidad de validación constante de la interfaz de usuario, se ha optado por un enfoque híbrido ágil: **Desarrollo Rápido de Aplicaciones (RAD)** complementado con el **Modelo de Prototipos**.

  • **RAD:** Prioriza una arquitectura de datos sólida desde el inicio (mediante el E-R) y permite iteraciones rápidas en la lógica de negocio (Alertas y Presentación).
  • **Prototipos:** Se utiliza para validar rápidamente los requisitos de la interfaz de usuario, especialmente para el Dashboard y la configuración avanzada de Alertas, asegurando la satisfacción del usuario final antes de invertir tiempo en la codificación completa del backend.

2.2. DFD Nivel 1 (Procesos Clave)

Descomponemos el sistema en sus tres procesos fundamentales:

  • **P1: Gestión de Portfolio (CRUD):** Permite al **Usuario** registrar, actualizar o eliminar sus **Transacciones** (D1).
  • **P2: Motor de Presentación:** Muestra el **Dashboard** al **Usuario**, calculando ganancias/pérdidas en tiempo real usando datos de **Transacciones** (D1) y **API de Precios**.
  • **P3: Motor de Alertas (Cron Job):** Proceso automatizado que se ejecuta periódicamente, consultando **Alertas** (D2) y **API de Precios** para generar **Notificaciones** al **Usuario** si se cumple una condición.

4. Modelización de Redes: Arquitectura 3-Tier Web

Para asegurar la **seguridad** y **eficiencia**, se ha elegido una arquitectura de tres capas (3-Tier):

Capa 1 (Cliente): Navegador Web (Presentación - HTML/CSS/JS/React)

Capa 2 (Servidor): Servidor de Aplicaciones (Lógica de Negocio - API REST, Motores P2 y P3)

Capa 3 (Datos): Servidor de Base de Datos (MongoDB - Almacenes D1 y D2)

**Justificación:** El aislamiento de la Base de Datos (Capa 3) del acceso directo del Cliente (Capa 1) maximiza la seguridad. Toda petición pasa por la Capa 2, que valida y gestiona la lógica, garantizando el control sobre las transacciones y la ejecución del Motor de Alertas.
